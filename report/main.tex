%%------------------摘要-------------%%
\begin{abstract}

本课程设计基于TI TM4C1294NCPDT (S800开发板) 设计并实现了一套\textbf{智能联网时钟系统}。MCU端独立运行本地时钟、闹钟、编辑状态机、流水显示等全部核心功能，通过UART 115200 8N1串口协议与PC端PyQt5数字孪生面板通信。PC端1:1镜像MCU数码管、LED、按键状态，并提供完整的P2控制面板（7个标签页）、NTP网络对时(E1)、天气获取(E2)、自动昼夜模式(E3)、数据可视化(E4)以及多闹钟调度器等扩展功能。实现了贪吃蛇小游戏作为自定义创新的MCU+PC联动功能。

系统通过了100条自动化协议测试，TIMEOUT=0，所有功能正常运行。本报告详细阐述了系统架构设计、MCU端硬件驱动与功能模块、PC端数字孪生与控制面板、串口通信协议设计、ARMCC5编译器适配与优化历程，以及完整的测试验证过程。

\end{abstract}

\subsection{I2C数码管显示}

\subsubsection{硬件设计}

7段数码管通过TCA6424 I2C扩展器（地址0x22）驱动：Port1输出段码数据（含dp位），Port2输出位选信号。段码表支持50个字符：0-9、A-Z（全部26字母）、小写c/e/h/i/j/l/n/o/r/u、符号- \_ . °。

\subsubsection{消鬼影刷新}

采用三段式刷新策略消除鬼影：全灭 → 加载段码数据 → 选通当前位。刷新频率约500Hz（每位2ms × 8位）。

\subsubsection{FORMAT RIGHT与小数点跟随}

在RIGHT模式下，字符串逆序显示，小数点dpHex需重新计算。关键代码逻辑：
\begin{verbatim}
if (g_format == FMT_RIGHT) {
    // 字符串逆序
    for (i = 0; i < DISP_LEN; i++)
        rev[i] = str[DISP_LEN - 1 - i];
    dp = (1 << 4) | (1 << 2);  // dp重新计算
    Display_SetStr(rev, dp);
} else {
    dp = (1 << 6) | (1 << 4);  // LEFT: 本位dp
    Display_SetStr(str, dp);
}
\end{verbatim}

\subsection{按键扫描}

\subsubsection{统一状态机}

按键扫描采用统一的KS\_IDLE → KS\_DEBOUNCE\_DOWN(20ms) → KS\_PRESSED → KS\_LONG\_PRESSED(800ms) 状态机。I2C按键（K0-K7）和GPIO按键（USER1/USER2）统一纳入uint16\_t all\_raw（低8位I2C + bit8/9 GPIO），共享消抖和长按逻辑。

\subsubsection{KEV\_UP消抖释放}

新增KS\_DEBOUNCE\_UP状态等待20ms确认释放后才推送KEV\_UP事件，过滤I2C总线毛刺导致的"假释放"，消除滚动模式下按键释放误退出的问题。

\subsection{编辑状态机}

FUNC短按循环切换：EDIT\_DATE → EDIT\_TIME → EDIT\_ALARM →（回到CLOCK可通过SAVE/FUNC长按退出）。SHIFT切换编辑字段（6位 × 2位一组），ADD当前字段+1（带范围钳制）。5秒无操作自动退出且不保存。\textbf{备份恢复机制}：进入编辑时备份当前值（edit\_backup\_time/date/alarm），FUNC模式切换保留编辑内容，仅退出时恢复备份。

\subsection{流水显示}

Scroll\_Tick每250ms（快）/ 500ms（慢）移动一个字符。SPEED键2级速度切换，滚动途中可立即生效。FORMAT键切换LEFT/RIGHT方向，小数点按照"本位"或"下一位"规则跟随。大于8字符自动入流水完整一遍退出，小于等于8字符静态显示2.5秒后自动退出。

\subsection{闹钟与蜂鸣器}

\subsubsection{PWM硬件驱动}

蜂鸣器采用PK5引脚M0PWM7，PWM0 Gen3 Out7输出2kHz/50\%占空比方波：

\begin{verbatim}
PWMGenPeriodSet(PWM0_BASE, PWM_GEN_3, 8000);     // 2kHz
PWMPulseWidthSet(PWM0_BASE, PWM_OUT_7, 2000);     // 50% duty
\end{verbatim}

\subsubsection{闹钟响铃相位切换}

闹钟触发后PWM始终运行，通过PWMOutputState(true/false)零延迟开关输出实现300ms响/300ms停的对称音效。采用绝对时间取模而非除法避免相位泄漏：

\begin{verbatim}
uint8_t should_be_on = ((elapsed % 600) < 300) ? 1 : 0;
if (should_be_on != beep_phase_on) {
    beep_phase_on = should_be_on;
    if (beep_phase_on) Beep_OutOn(); else Beep_OutOff();
}
\end{verbatim}

\subsubsection{持久静音Watchdog}

远程蜂鸣超时后设置beep\_force\_off=1，每100ms循环执行Beep\_Off()，确保PWM计数器即使停在HIGH相位也强制拉低PK5输出直至GPIO电平为0。*RST等复位操作不会取消watchdog，反而加强。

此设计的核心难点在于\underline{区分两种响鸣需求}：闹钟需快速响应(300ms切换)而BEEP需绝对静音，以及\underline{PWM计数器随机停在高电平}的硬件不可控性。

\subsubsection{双路径蜂鸣驱动（关键代码）}

\begin{verbatim}
/* BEEP / 闹钟启动 */
static void Beep_On(void)  { PWMGenEnable(PWM0_BASE, PWM_GEN_3); }
static void Beep_Off(void) { PWMGenDisable(PWM0_BASE, PWM_GEN_3); }

/* 闹钟相位切换 (PWM始终运行, 仅开关输出, 零延迟) */
static void Beep_OutOn(void)  { PWMOutputState(PWM0_BASE, PWM_OUT_7_BIT, true); }
static void Beep_OutOff(void) { PWMOutputState(PWM0_BASE, PWM_OUT_7_BIT, false); }

/* 闹钟: Beep_On启动 → 循环Beep_OutOn/Off(300ms对称) → Alarm_Stop→Beep_Off */
/* BEEP:  Beep_On → 等Nms → Beep_Off + beep_force_off持久watchdog            */
\end{verbatim}

\section{串口通信协议设计}

\subsection{协议概述}

\begin{itemize}
    \item 物理层: UART 115200, 8N1, 无流控
    \item 行格式: 命令以\textbackslash r\textbackslash n结束
    \item 大小写: \textbf{不敏感}（ExtractLine转大写）
    \item 空格: 非MSG命令丢弃所有空格（紧邻token流）
    \item 缩写: cmd\_match前缀匹配 + skip\_kw\_rest后缀跳过（如MIN→MINUTE跳过"UTE"）
    \item 错误: ERROR SYNTAX / PARAM / RANGE / LINE TOO LONG
\end{itemize}

\subsection{UART接收与行整理}

UART RX ISR将接收字节写入256字节环形缓冲区，遇\textbackslash r/\textbackslash n置位rx\_line\_ready。主循环中ExtractLine()进行行整理：

\begin{enumerate}
    \item 跳过buffer中残留的\textbackslash r/\textbackslash n
    \item 窥探前15字节判断是否为*SET:MSG（保留大小写+空格）
    \item 非MSG命令：丢弃空格 + 小写转大写 → 紧邻token流
    \item 行长>64字节丢弃并回复ERROR LINE TOO LONG
\end{enumerate}

\subsection{命令解析架构：2-Pass内联解析}

针对ARMCC5 C89编译器的优化缺陷，DATE/TIME/ALARM三个handler采用创新的2-pass解析架构：

\begin{enumerate}
    \item \textbf{Pass 1}：副本指针char *q扫描关键词 → cmd\_match前缀匹配 → 建立kmap bitmask（记录哪些字段被指定）
    \item \textbf{Pass 2}：原始指针char *p逐字符扫描 → 内联while(*p>='0')\{v=v*10+(*p++-'0')\}提取整数 → vals[]
    \item \textbf{Map}：按kmap顺序分配vals到目标变量 + 默认值回填
\end{enumerate}

\textbf{关键设计决策}：所有数字解析使用自写内联循环，\textbf{零次strtol调用}，从根本上消除了ARMCC5 C89运行时库strtol(t, \&t, 10)中\&t写回偏移的缺陷。

\subsection{2-Pass解析核心代码}

ExtractLine去掉空格后的关键效果: "*SET:TIME HOUR MIN SEC 14 30 00" → "*SET:TIMEHOURMINSEC143000"，解析变为连续的逐字符扫描。

\begin{verbatim}
/* Pass 1: copy pointer scan keywords → kmap bitmask */
q = p;  kmap = 0;
while (*q) {
    if (cmd_match(q, "HOUR"))  { kmap |= 1; q += 4; skip_kw_rest(&q, ""); }
    else if (cmd_match(q, "MIN")) { kmap |= 2; q += 3; skip_kw_rest(&q, "UTE"); }
    else if (cmd_match(q, "SEC")) { kmap |= 4; q += 3; skip_kw_rest(&q, "OND"); }
    else break;
}  // ← 遇到未知字母 → ERROR SYNTAX

/* Pass 2: inline digit scan → vals[] (no strtol!) */
wi = 0;
while (*p && wi < 3) {
    if (*p >= '0' && *p <= '9') {
        int v = 0;
        while (*p >= '0' && *p <= '9' && dc < 2) { v=v*10+(*p++-'0'); dc++; }
        vals[wi++] = v;
    } else { p++; }
}

/* Map: kmap order → h/m/s + default fill */
h=-1;m=-1;s=-1; vi=0;
if (kmap & 1) h = (vi<wi) ? vals[vi++] : -1;
if (kmap & 2) m = (vi<wi) ? vals[vi++] : -1;
if (kmap & 4) s = (vi<wi) ? vals[vi++] : -1;
while (vi<wi) { if(h<0)h=vals[vi]; else if(m<0)m=vals[vi]; else if(s<0)s=vals[vi]; vi++; }
\end{verbatim}

\noindent\underline{设计要点}：cmd\_match使用大小写敏感的前缀匹配（ExtractLine已将输入转大写），每次调用后指针跳过关键词长度并调用skip\_kw\_rest吞掉可选的缩写后缀。数字扫描严格限制位数上限（TIME/ALARM 2位、DATE的YEAR 4位），防止去空格后多字段数字黏连。

\section{技术亮点与难点突破}

\subsection{亮点1: ExtractLine去空格+2-Pass架构 （ARMCC5适配关键）}

本项目最具代表性的技术决策。面对ARM Compiler 5 C89的寄存器缓存缺陷，常见思路是修改代码绕过编译器bug。但我们选择了从架构层面消除缺陷路径：\underline{去掉命令中的所有空格}，使解析指针不再需要跳过空格→关键词和数字紧邻→只需直行逐字符扫描。配合自写的内联数字解析替代strtol，从根本上避开了ARMCC5在"指针偏移→二级指针写回"路径上的全部缺陷。

该方案同时带来了额外收益：解析器支持全关键词前置和关键词-值交错两种命令格式，且代码量减少了约240行。

\subsection{亮点2: 双硬件路径蜂鸣器驱动}

蜂鸣器的闹钟响铃和远程BEEP同时正确驱动，需要解决一个硬件层面矛盾：闹钟需要300ms快速开关切换（PWM不能启停延时），BEEP需要绝对静音保证（PWM关断后引脚必须低电平）。采用\textbf{双函数族方案}：Beep\_On/Off用GenEnable/Disable为BEEP提供彻底关断；Beep\_OutOn/Off用OutputState(true/false)为闹钟提供零延迟开关。配合beep\_force\_off持久静音watchdog（100ms循环Beep\_Off），确保极端情况下也彻底静音。

\subsection{亮点3: PC虚拟面板1:1数字孪生}

PC端不依赖任何第三方7段数码管控件，采用QPainter从零自绘50种字符（含全部26字母、小写、符号、小数点独立占位）。*EVT:DISP每秒上报8字符+dpHex位图，PC端100\%镜像MCU木板显示状态。独立的8位LED控件用QSS渐变圆点渲染，10键虚拟按键支持PC→MCU双向交互。NIGHT模式下仅显示前4位(HH.MM)，编辑高亮同步闪烁。

\subsection{难点1: ARMCC5编译器适配} 耗时约40小时。问题概率性触发、随代码修改转移位置，经历了5轮不同策略的尝试才找到架构级解决方案。

\subsection{难点2: 蜂鸣器稳定性} 涉及PWM硬件行为与软件同步的复杂交互。PWMGenDisable后计数器停在随机相位是根本原因，需要OutputState+watchdog双保险。

\subsection{难点3: I2C总线竞态} SysTick ISR与前台代码共享I2C总线，需要SysTickIntDisable+while(I2CMasterBusy)等待in-flight事务后安全访问。

\section{ARMCC5编译器适配与UART解析Bug修复}

\subsection{问题现象}

ARM Compiler 5 V5.06在C89模式下编译大型多分支函数时，优化器产生不正确的寄存器分配，导致两个关键问题：

\begin{itemize}
    \item \textbf{死循环}: while(*t)和while(*msg)等裸指针循环中，*t值被缓存在寄存器，循环体内指针递增不写回寄存器→永远非零→MCU卡死
    \item \textbf{数据截断}: strtol(t, \&t, 10)库函数通过二级指针\&t写回更新后的指针值，但ARMCC5将该写回偏移了一个字节，导致下次迭代从错位地址读取→14→4、30→0等截断
    \item \textbf{概率性触发}: 缺陷是否触发依赖特定的栈帧布局。添加或删除任何代码都会改变布局→缺陷位置和表现随之转移，难以精确定位
\end{itemize}

\subsection{核心解决方案：ExtractLine空格清除 + 内联数字解析}

经过持续调试与分析，我们发现了问题的根本性质：传统解析流程中，命令字符串包含空格，解析指针需要不断跳过空格后再读关键词和数字，这需要复杂的指针偏移操作——而ARMCC5恰好在这种场景下产生了寄存器缓存错误。解决思路不是"与编译器对抗"（如volatile强制重载），而是\textbf{从架构层面消除指针偏移的需求}：

\begin{enumerate}
    \item \textbf{ExtractLine去空格}：将行整理函数改为丢弃所有非MSG命令的空格，输入 "*SET:TIME HOUR MIN SEC 14 30 00" → "*SET:TIMEHOURMINSEC143000"。这从根本上消除了关键词和数字之间的分隔符——参数紧邻关键词，无需strspn/strcspn跳过空格。
    \item \textbf{cmd\_match前缀匹配}：由于ExtractLine已将命令转为大写、去除空格，解析器不再需要处理空格。cmd\_match对紧凑token流进行大小写不敏感的前缀匹配（如MIN匹配MINUTE的前三个字符），配合skip\_kw\_rest跳过缩写后缀。
    \item \textbf{2-Pass架构}：Pass1用副本指针扫描关键词建kmap，Pass2用原指针逐字符扫描提取数字。由于没有空格，Pass2的数字提取只需要一个简单的内联while循环。
    \item \textbf{内联数字解析}：所有数字提取改用自写的while(*p>='0')\{v=v*10+(*p++-'0')\}，不使用strtol——没有二级指针写回，彻底避开了ARMCC5的那条缺陷路径。
\end{enumerate}

\subsection{辅助措施}

\begin{itemize}
    \item \textbf{\#pragma O0}: 文件级禁用编译器优化，作为全局安全网，确保所有函数的代码生成都是确定性的。编译产物体积略大但不影响功能
    \item \textbf{volatile关键变量}: remote\_beep\_active、remote\_beep\_end\_ms等可能被ISR异步修改的变量声明为volatile，保证每次读取都从内存重载
    \item \textbf{cmd\_line栈拷贝}: ProcessCommand前将全局cmd\_line拷贝到栈局部变量，进一步隔离UART ISR写入对解析过程的潜在干扰
    \item \textbf{UART输出函数修复}: 将while(*msg)改为for+strlen计数循环，消除UART发送中的同一类死循环风险
\end{itemize}

\section{开发反思与经验总结}

\subsection{UART通信不稳定与ARMCC5编译器适配 \qquad (\faStar\,\faStar\,\faStar 最高难度)}

本项目耗时最长（约40小时）、定位最困难的问题。\textbf{难度评价: ★★★★★}。现象表现为：执行自动化测试时，MCU偶尔卡死（无任何响应）或SET命令解析出的数值出现截断（如14→4, 30→0）。问题概率性出现，且随着代码修改位置变化而转移。

\subsubsection{问题表现}

\begin{enumerate}
    \item \textbf{卡死}: 执行*RST紧接*SET:TIME后，MCU完全无响应，PING无应答
    \item \textbf{截断}: *SET:TIME HOUR 14 MIN 30 SEC 00 → *GET:TIME返回04.00.00（HOUR=4而非14）
    \item \textbf{概率性}: 有时连续10次测试正常，有时第2次即触发，难以复现
    \item \textbf{转移性}: 添加debug标记后问题位置改变（卡死→截断），删除标记后截断仍存
\end{enumerate}

\subsubsection{诊断过程}

\begin{enumerate}
    \item \textbf{初始怀疑——指针while循环}: 原始代码使用while(*t)和while(*msg)等裸指针循环。ARMCC5 C89模式下，编译器将*t的值缓存在寄存器中，循环体内t++的更新未写回寄存器→死循环。修复：UART输出函数改为for+strlen计数循环。

    \item \textbf{添加debug标记}: 为精确定位卡死位置，在ProcessCommand每个阶段设置g\_dbg标记(0x01-0xF1)。通过Keil调试器暂停后查看g\_dbg值定位卡死点。

    \item \textbf{发现截断问题}: debug标记添加后卡死消失，但出现新的数据截断。通过*DBG诊断行上报kmap和vals[]值，发现kmap=7(关键词全部正确识别)但vals[1]=-1或vals[0]被截断——即关键词识别正确、数字提取错误。反复验证后确认根因是ARMCC5 C89运行时库的strtol(t, \&t, 10)在通过二级指针写回时偏移了一个字节。

    \item \textbf{排除debug标记干扰}: 逐一删除ProcessCommand内、ExtractLine内、PutStrNB内的所有volatile调试标记，确认截断与标记本身无关。

    \item \textbf{尝试volatile与\#pragma O0}: 使用文件域volatile指针替代局部变量、添加\#pragma O0禁用优化。卡死问题解决，截断大幅减少但未完全消除——说明库函数strtol本身的缺陷与优化与否无关，已编译在.lib中的代码无法通过编译器开关修正。

    \item \textbf{架构级根除——ExtractLine去空格 + 内联解析}: \textbf{这是解决问题的关键转折点}。既然ARMCC5在处理"空格→指针偏移→strtol二级指针写回"这条路径上存在缺陷，最根本的方案不是修补，而是\textbf{从架构层面消除这条路径}：
    \begin{itemize}
        \item \textbf{ExtractLine去空格}：将非MSG命令的空格全部丢弃，使关键词和参数紧邻（例如 *SET:TIME HOUR MIN SEC 14 30 00 → *SET:TIMEHOURMINSEC143000）。这从根本上消除了解析过程中需要指针跳过空格的复杂偏移操作
        \item \textbf{内联数字解析}：所有数字提取改为自写while(*p>='0')\{v=v*10+(*p++-'0')\}，不使用strtol——没有二级指针写回，彻底避开该缺陷路径
    \end{itemize}
    \textbf{关键洞察}: 空格清除后p指针只需"直行"逐字符扫描，而非在关键词、空格、数字之间来回跳跃偏移。解析逻辑变得线性直接，ARMCC5无法产生缺陷。

\end{enumerate}

\subsubsection{最终解决方案}

\begin{itemize}
    \item \textbf{ExtractLine去空格}: 所有非MSG命令丢弃空格，命令转为大写紧邻token流 → \textbf{从根本上消除指针偏移操作}
    \item \textbf{内联数字解析}: while(*p>='0')\{v=v*10+(*p++-'0')\}，零次strtol调用 → 避开ARMCC5运行时库缺陷
    \item \textbf{2-Pass解析}: Pass1扫关键词建kmap，Pass2扫数字填vals，按kmap映射 → 处理全关键词和关键词-值交错两种格式
    \item \textbf{\#pragma O0}: 文件级禁用优化，作为全局安全网确保所有代码生成确定
    \item \textbf{cmd\_line栈拷贝}: ProcessCommand前将全局buffer拷贝到栈变量，隔离UART ISR写入干扰
\end{itemize}

\subsection{蜂鸣器稳定性问题 \qquad (\faStar\,\faStar\,\faStar 高难度)}

\subsubsection{从PN1到PK5的迁移}

最初使用PN1引脚通过Timer0A ISR软件翻转2kHz波形。烧录后发现蜂鸣器完全不响。检查硬件原理图后确认S800板的蜂鸣器实际连接在PK5引脚，驱动方式应为M0PWM7/PWM0 Gen3 Out7硬件PWM。

\subsubsection{闹钟响铃高低不稳}

迁移到PK5 PWM后，蜂鸣器能响但声音明显一高一低。根因：PWMGenDisable停止计数器后，PWM计数器可能停在HIGH相位→引脚锁在高电平→连续尖锐鸣叫。修复过程：

\begin{enumerate}
    \item 尝试Beep\_Off后GPIO强制拉低PK5 → 部分有效
    \item 尝试Beep\_Off先PWMOutputState(false)再GenDisable → 改善但仍有偶发
    \item 最终方案——\textbf{持久静音watchdog}: beep\_force\_off标志位，超时后每100ms循环执行Beep\_Off()，直到下次Beep\_On()清零
\end{enumerate}

\subsubsection{*RST打断静音}

testcmd.txt中*SET:BEEP 1000紧跟*RST。BEEP超时后beep\_force\_off=1启动持久静音，但*RST中beep\_force\_off=0把watchdog杀了。修复：*RST改为Beep\_Off()+beep\_force\_off=1，不取消watchdog。

\subsubsection{闹钟不跟随BEEP同时Fix}

闹钟响铃需要快速切换（300ms响/300ms停），BEEP需要彻底关断。两条需求矛盾——GenEnable/Disable有延时保证静音但不适合快速切换，PWMOutputState零延迟但不保证彻底关断。最终采用\textbf{双路径方案}：

\begin{itemize}
    \item \textbf{闹钟}: Beep\_On启动PWM → 相位循环中Beep\_OutOn/OutOff用OutputState快速切换 → Alarm\_Stop用Beep\_Off彻底关断
    \item \textbf{BEEP}: Beep\_On → 等Nms → Beep\_Off + 持久静音watchdog
\end{itemize}

\subsection{按键扫描演进 \qquad (\faStar\,\faStar 中高难度)}

\subsubsection{SPEED/FORMAT滚动中途误退出}

按键释放时Key\_Scan推送KEV\_UP事件。SCROLL状态下SPEED/FORMAT的KEV\_DOWN正确处理，但其后的KEV\_UP不匹配SPEED/FORMAT的KEV\_DOWN条件→落入else"任意键退出"分支→滚动被意外终止。修复：整个SCROLL分支用ke==KEV\_DOWN包裹。

\subsubsection{统一状态机重构}

原始代码中I2C按键(8键)和GPIO按键(USER1/USER2)使用两套独立的处理逻辑——I2C走状态机、GPIO裸边沿检测。这导致USER1/USER2无消抖保护、无长按支持。重构为：all\_raw=uint16\_t(I2C8位+GPIO2位)→统一KS\_IDLE→KS\_DEBOUNCE\_DOWN(20ms)→KS\_PRESSED→KS\_LONG\_PRESSED(800ms)→KS\_DEBOUNCE\_UP(20ms释放消抖)→KS\_IDLE。

\subsubsection{KEV\_UP消抖释放}

新增KS\_DEBOUNCE\_UP状态：释放后等待20ms确认无新按键→推送KEV\_UP→回KS\_IDLE。中途再按则转回KS\_PRESSED。过滤I2C总线毛刺导致的"假释放"。

\subsection{命令解析架构演进 \qquad (\faStar\,\faStar 中等难度)}

\subsubsection{从match\_abbrev到cmd\_match}

原始方案使用match\_abbrev函数实现缩写匹配（大写必输、小写可选），通过混合大小写模式(MINute, SECond)标记哪些字母可省略。但这个设计中，全大写模式(MONTH)禁止缩写，而MINute要求至少输MIN。测试中出现非法缩写(MONT, MI)通过的问题。改为cmd\_match前缀匹配：MONT vs MONTH→M-O-N-T匹配后input耗尽但prefix='H'≠0→失败。MI vs MIN→M-I匹配后input耗尽但prefix='N'≠0→失败。

\subsubsection{从单Pass到2-Pass}

ExtractLine去空格后*SET:TIME HOUR MIN SEC 14 30 00→HOURMINSEC143000。单pass cmd\_match+has\_digit的分支设计无法处理全关键词在前、值全在末尾的格式。改用2-Pass：Pass1用副本指针q扫关键词→建kmap，Pass2用原指针p扫所有数字→vals[]，Map阶段按kmap顺序分配。

\subsection{其它修复记录}

\begin{itemize}
    \item \textbf{PC端1Hz自动PING+3s超时}: 通过QTimer 1000ms发*PING，超过3s无*PONG→状态灰变+自动重试
    \item \textbf{LED位映射修正}: LED3=UART TX+RX合并(0x08), LED4=SUN(0x10), LED5=RAI/SNO(0x20), LED6=高温(0x40), LED7=NTP(0x80)
    \item \textbf{编辑备份恢复}: 进入编辑时memcpy备份→FUNC短按模式切换保留编辑→SAVE/FUNC长按提交→退出时仅恢复未保存的值
    \item \textbf{MSG前导空格}: skip\_kw\_rest后p指向" Hello"→while(*p==' ')p++跳过空格
    \item \textbf{PC闹钟管理崩溃}: \_update\_alarm\_ui引用不存在的btn\_alarm\_toggle→改为no-op; 每行新增设置/清除按钮
    \item \textbf{闹钟DOW slot解析}: ALARM handler 2-pass之前加dow\_to\_slot()检测MON-SUN，匹配到则覆盖ss/se，无DOW默认当天
    \item \textbf{参数组合演示}: PC日期时间页QComboBox 4种组合(YEAR MONTH DATE等)+缩写演示按钮发送cmd\_match可接受的简称
\end{itemize}

\section{总结与展望}

\subsection{项目成果}

本课程设计完成了一套功能完整的智能联网时钟系统：
\begin{itemize}
    \item MCU端：独立运行的本地时钟（含闰年）、闹钟（节奏响铃+天气联动）、流水显示（双向双速）、编辑状态机（备份恢复）、按键扫描（统一状态机消抖）、LED指示（8位精确映射）、15条UART命令+8类事件解析
    \item PC端：7标签页控制面板、数字孪生镜像面板（1:1同步）、4项扩展功能、贪吃蛇游戏
    \item 协议：100条自动化测试通过，TIMEOUT=0
    \item 编译器：成功适配ARMCC5 C89优化缺陷，\#pragma O0 + 内联解析双保险
\end{itemize}

\subsection{技术难点与核心收获}

\begin{enumerate}
    \item \textbf{ARMCC5编译器适配}: 通过ExtractLine去除空格+内联解析的架构级方案，从根源上消除了指针偏移操作→ARMCC5寄存器缓存缺陷的触发路径。配合\#pragma O0和cmd\_line栈拷贝，彻底解决了死循环和数据截断两大问题。深入理解了C89编译器的行为特性
    \item \textbf{I2C总线竞态}: SysTick ISR与前台代码共享I2C → SysTickIntDisable+while(I2CMasterBusy())等待in-flight事务
    \item \textbf{蜂鸣器稳定性}: PWM计数器随机停在高电平 → 先OutputState(false)再GenDisable + beep\_force\_off持久静音 + 闹钟/BEEP双路径驱动
    \item \textbf{按键扫描}: 统一状态机 + KS\_DEBOUNCE\_UP消抖释放 → 根除KEV\_UP误触发导致的各种异常
\end{enumerate}
